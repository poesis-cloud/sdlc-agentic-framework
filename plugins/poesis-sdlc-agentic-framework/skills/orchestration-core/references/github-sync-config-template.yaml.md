# Per-product sync config template — `portfolio/<slug>/github-sync.yaml`

One config per product. Generated by `portfolio/_sync/sync.py init <slug>` from
`portfolio/_registry.yaml` + this template, then committed. The board spec
([github-projects-board-spec.md](./github-projects-board-spec.md)) is the normative schema;
this file binds one product to its Project. Do **not** hand-edit the `project.number` /
`project.id` once `provision` has filled them.

```yaml
# portfolio/<slug>/github-sync.yaml
org: poesis-cloud
product: <slug>                      # matches portfolio/_registry.yaml

project:
  title: "Poesis — <Name>"          # board spec §2
  number: null                       # filled by `provision`
  id: null                           # ProjectV2 node id, filled by `provision`

planning_repo: <slug>-planning       # under org; issues-only, NOT a code repo

issue_types:                         # org-level Issue Types (board spec §3)
  feature: Feature
  story: Story

paths:                               # globs relative to portfolio/<slug>/
  features_glob: "features/F-*.md"
  stories_glob: "sprint-*/stories/S-*.md"

# Single-select fields that drive the two boards (board spec §4–§5).
status_fields:
  program: "Program Status"
  team: "Team Status"

status_options:
  program: [funnel, refined, arch-pending, ready, committed, in-progress, done]
  team:    [backlog, ready, in-progress, in-review, in-qa, awaiting-pr, done]

# Columns whose entry/exit crosses a human gate (board spec §8). Only the
# genuinely *awaiting-gate* columns are listed — they drive the `Gate` display
# field. The `→ done` crossing (★ PR Gate / ★ Demo Gate) is gate-protected independently
# by the sync (protocol §4), so `done` is intentionally NOT listed here.
gate_columns:
  program:
    arch-pending: gate-architecture
  team:
    awaiting-pr: gate-pr

# Conflict-resolution authority (protocol §3).
authority:
  content: local                     # title/body/WSJF/risk/… authored locally
  status: remote                     # non-gate board moves win on pull
  destructive: false                 # never delete/close remote from local

# `blocked` is an orthogonal label, not a column (board spec §7).
blocked_label: blocked
```

## Notes

- `project.number` / `project.id` and all resolved field/option ids live in
  `portfolio/<slug>/.sync-state.json` (generated; git-ignored or committed per product policy).
- `status_options` intentionally omit `blocked` — it is a label, applied orthogonally.
- Changing `authority.status` to `local` makes local frontmatter win status conflicts; the
  default `remote` reflects that humans move cards on the board. Gate-crossing transitions are
  exempt from both directions (protocol §4).
- **Issue representation is config-free.** The `<type>:<status>` status labels (board spec §3b),
  the org Issue Types (§3), the `SAFe *` generic field names (§4), and the `.github/ISSUE_TEMPLATE`
  forms (§9b) are all derived deterministically from `status_options` + the SAFe type — the sync
  creates/assigns them on `provision` / `push`; no extra config keys are needed.

## Portfolio variant — `portfolio/github-sync.yaml`

The portfolio config is the same shape with three differences (generated by
`sync.py init portfolio`):

```yaml
# portfolio/github-sync.yaml
org: poesis-cloud
product: _portfolio
project:
  title: "Poesis — Portfolio"
  number: null
  id: null
planning_repo: poesis-portfolio-planning
issue_types:
  epic: Epic                         # this Project holds Epics, not Features/Stories
paths:
  epics_glob: "epics/E-*.md"         # relative to portfolio/
status_fields:
  portfolio: "Portfolio Status"
status_options:
  portfolio: [funnel, reviewing, analyzing, portfolio-backlog, implementing, done]
gate_columns:
  portfolio:
    analyzing: gate-epic             # awaiting the ★ Epic Gate; `portfolio-backlog`/`done` protected independently
authority:
  content: local
  status: remote
  destructive: false
blocked_label: blocked
```

The Feature→Epic sub-issue link is cross-repo: a Feature Issue (in `<slug>-planning`) is linked
as a sub-issue of its Epic Issue (in `poesis-portfolio-planning`) by the product push, reading
the Epic's node id from `portfolio/.sync-state.json`.
